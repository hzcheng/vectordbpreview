# Phase 2 Lesson 1: 向量数据库在 AI 中的典型应用场景

## 1. 先回答一个根问题

### 向量数据库到底解决什么问题
一句话：

```text
当你已经把对象编码成向量后，需要一个系统在大量向量里高效地存、索引、查、过滤、更新。
```

所以它不是训练模型的，也不是生成 embedding 的。

它主要解决：
- 海量向量存储
- 高维近邻检索
- 标量过滤
- TopK 返回
- 实时或准实时更新
- 与业务 metadata 联动

## 2. 一条完整链路

最典型的 AI 检索链路是：

```text
原始数据
-> 清洗
-> 切分 chunk
-> embedding
-> 写入向量数据库
-> 查询向量生成
-> ANN 召回
-> metadata filter
-> rerank
-> 返回结果
```

### 每一步分别在干什么

#### 清洗
去掉噪声、空文本、重复内容、无意义字段。

#### 切分 chunk
把长文档切成适合 embedding 和检索的片段。

原因：
- 太长会语义混杂
- 太短又会丢上下文

#### embedding
把 chunk 编码成向量。

#### 写入向量数据库
存：
- id
- vector
- metadata
- 可选的原文或引用信息

#### 查询向量生成
把用户问题也编码成向量。

#### ANN 召回
先快速找一批“可能相关”的候选。

#### metadata filter
再按业务条件过滤，例如：
- 时间范围
- 租户 ID
- 文档类型
- 权限标签

#### rerank
用更贵但更准的模型重新排序。

#### 返回
给应用层返回文档、片段、分数和元数据。

## 3. 六大典型场景

### 场景 1: RAG
这是现在最常见的场景。

流程：
- 文档入库
- 文本切分
- embedding
- 用户提问
- 向量召回相关 chunk
- 把 chunk 交给 LLM 生成答案

向量数据库在里面的角色：
- 负责召回“和问题语义最相关”的上下文

它解决的问题：
- LLM 不知道你的私有知识
- 上下文窗口有限
- 需要从私有语料中动态取证

### 场景 2: 语义搜索
用户搜“怎么配置开发环境”，也能找到“Python 环境安装指南”。

它比关键词检索强的地方：
- 不依赖完全相同字面词
- 能找到表达不同但意思相近的内容

它的风险：
- 解释性比关键词差
- 容易召回“语义近但业务不对”的结果

### 场景 3: 推荐与召回
商品、内容、用户都可以编码成向量。

应用：
- 相似商品推荐
- 相似内容推荐
- 用户兴趣召回

向量数据库在这里主要干：
- 从亿级候选里先召回一批候选
- 后续再交给排序模型做精排

### 场景 4: 多模态检索
例如：
- 文搜图
- 图搜图
- 图搜文

前提：
- 不同模态被映射到同一个或可对齐的向量空间

例子：
- 搜“红色跑车”，返回图片
- 上传一张鞋子图片，找相似商品

### 场景 5: 去重、聚类、异常发现
如果语义接近的内容向量也接近，就可以：
- 找重复文档
- 找相似工单
- 做主题聚类
- 发现与已有样本差异很大的异常点

### 场景 6: 召回与重排分层架构
这是工程上非常关键但最容易忽视的一点。

常见做法不是：
- 直接一次得到最终最优答案

而是：
- 第一层用向量数据库做高速召回
- 第二层用 reranker 或业务模型做高精度重排

也就是：

```text
fast recall first, accurate ranking second
```

## 4. 什么时候适合用向量数据库

适合：
- 搜索目标与“语义相似”强相关
- 数据是文本、图片、音频、用户行为等非结构化或半结构化对象
- 数据量大到需要 ANN
- 召回结果允许概率式近似
- 需要和 metadata filter 一起工作

## 5. 什么时候不一定要用

不适合或不必优先：
- 完全精确匹配的查询
- 规则检索明显更简单的场景
- 数据量很小，普通数据库全表扫都足够
- 业务需要强解释性且关键词匹配足够有效
- 你还没有稳定的 embedding 方案

一句话：

```text
如果问题本质不是“找语义相近”，就不要硬上向量数据库。
```

## 6. 向量数据库不是单独工作的

一个成熟检索系统常常是混合检索：
- 向量召回
- 关键词召回
- metadata filter
- 规则打分
- reranker

原因：
- 向量擅长语义
- 关键词擅长精确词面
- filter 擅长业务约束
- reranker 擅长最终排序

## 7. 你从数据平台视角必须关注的点

### 数据模型
至少要想清楚：
- 主键是什么
- 一个对象是一个向量还是多个向量
- metadata 存什么
- 原文是否单独存储

### 更新策略
要想清楚：
- 原文更新后是否重算 embedding
- 是否允许部分更新
- 是否批量重建索引

### 质量评估
你不能只看“检索速度”。
还要看：
- 召回率
- Precision
- 业务命中率
- rerank 后效果

### 成本控制
你还要考虑：
- 向量维度
- 索引构建成本
- 存储成本
- 实时写入和查询并发

## 9. 一个面向你自己的简化判断框架

遇到一个新需求时，先问 4 个问题：

1. 这个问题本质是不是“找语义相近”？
2. 数据规模是否已经需要 ANN？
3. 是否需要 metadata filter 和多租户隔离？
4. 是否需要召回后再做更精细的排序？

如果这四个问题里有三个是“是”，大概率值得引入向量数据库。
