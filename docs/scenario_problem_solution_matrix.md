# 向量数据库场景-问题-方案对照表

## 目标
用一页矩阵快速回答三个问题：
- 这类场景到底在解决什么问题
- 向量数据库在链路里负责哪一段
- 从平台接入视角，应该优先设计哪些能力

| 场景 | 典型问题 | 向量数据库的角色 | 常见方案形态 | 不适合或主要风险 | 平台侧优先关注点 |
|------|----------|------------------|--------------|------------------|------------------|
| RAG | 用户问题和知识文档字面不一致，纯关键词难召回相关上下文 | 负责把问题向量映射到相关 chunk，完成候选上下文召回 | `chunk -> embedding -> vector recall -> metadata filter -> rerank -> LLM` | chunk 切分不合理、embedding 质量差、召回结果缺少权限过滤 | chunk 粒度、文档版本、权限标签、embedding 更新、引用回溯 |
| 语义搜索 | 用户搜索“开发环境配置”，但文档标题写的是“Python 环境安装指南” | 负责语义近邻召回，弥补关键词不一致 | `keyword recall + vector recall + merge/rerank` | 解释性较弱，容易召回“语义近但业务不对”的结果 | 查询日志、召回评估、metadata filter、混合检索编排 |
| 推荐/召回 | 需要从海量商品、内容、用户行为中快速找相似候选集 | 负责第一阶段候选召回，不负责最终业务排序 | `item/user embedding -> ANN recall -> ranker` | 把向量召回误当最终排序，忽略实时特征和业务规则 | 主键模型、在线特征更新、召回与精排边界、批量重建策略 |
| 多模态检索 | 文搜图、图搜图、图搜文，需要跨模态相似匹配 | 负责存储和检索对齐后的跨模态向量 | `multimodal embedding -> vector recall -> filter -> rerank/display` | 模态对齐不稳定，metadata 设计不足会影响过滤和解释 | 多向量/多字段 schema、模态标识、原始资源引用、过滤字段 |
| 去重/聚类/异常发现 | 需要发现语义重复内容、相似工单或偏离样本分布的异常数据 | 负责近邻查找和相似度计算基础能力 | `embedding -> KNN / threshold match / clustering pipeline` | 阈值难定，业务上“相似”不一定等于“重复” | 阈值配置、批处理任务、结果审计、人工校正回流 |
| 召回 + 重排 | 既要低延迟，又要较高结果质量 | 负责快速召回候选，给后续 reranker 缩小范围 | `vector recall + keyword recall + business rules + reranker` | 误把数据库当成完整检索系统，导致召回和排序职责混淆 | TopK 策略、候选集大小、重排接口、延迟预算、评估闭环 |

## 简化判断框架

如果一个需求同时满足下面至少 3 条，通常值得进入向量数据库设计阶段：

1. 问题本质是“找语义相近”，不是精确匹配。
2. 数据规模已经大到需要近似近邻检索。
3. 需要和 `metadata filter`、权限控制、多租户边界一起工作。
4. 系统架构天然需要“召回”和“重排”分层。

## 对 Phase 4A 的直接输入

进入平台接入草图前，至少要先写清楚这 5 个问题：

1. 我的平台里“被向量化的对象”到底是什么，是整条记录、字段片段还是文档 chunk。
2. 主键和版本号如何设计，才能支持增量更新和重算 embedding。
3. 哪些 metadata 需要进入向量库，哪些应只保留在主存储。
4. 检索链路里，向量召回、过滤、重排、权限校验分别在哪一层执行。
5. 哪些场景要单向量，哪些场景可能要多向量或多模态。
