# Milvus 7-Theme Learning Design

## Context
当前项目已经完成了三层前置积累:

1. 向量与检索基础
2. 向量数据库应用场景
3. 一个最小 Milvus Lite Demo

同时也完成了 Milvus-first 主线下的三个中间研究产物:

- 版本与术语基线
- 混合数据模型
- 存储组织与索引承载关系

用户的新目标不是继续按原有 `Phase D / Phase E` 的研究顺序推进，而是改成一条更适合学习与复盘的主题驱动主线。

## User Goal
用户希望一步一步从概念到代码，系统熟悉整个 Milvus，重点覆盖:

1. 相关概念
2. 整体架构
3. 数据写入流程
4. 向量数据和其他数据的组织格式
5. 不同数据索引的组织方式
6. 查询架构
7. 查询流程串联

用户同时明确要求:

- 严格按这 7 个主题推进
- 一题一文档
- 每讲完一个主题，都要直接映射到本地源码目录
- 每份文档都要包含关键目录、关键文件、建议阅读顺序
- 本地源码树以 `/Projects/work/vectordbpreview/milvus` 为准

## Recommended Approach
采用“主题驱动 + 源码并行映射”的学习方式。

每个主题都固定按以下结构输出:

1. 这个主题在 Milvus 里解决什么问题
2. 最小概念集合与心智模型
3. 与前后主题的关系
4. 本地源码目录映射
5. 关键文件入口
6. 建议阅读顺序
7. 当前主题文档落盘

这种方式优于“先讲完所有主题再统一看源码”，因为它能在概念刚被理解时立即建立源码映射，减少后续重复建模成本。

## Scope Boundaries

### In Scope
- 用 Milvus `2.6.x / v2.6.7` 作为稳定研究基线
- 使用官方文档与本地源码树联合讲解
- 形成 7 份主题文档
- 把主题理解落到本地源码目录与关键文件

### Out Of Scope
- 当前不继续推进 `notes RAG` 平台设计
- 当前不推进 RedDB 对接
- 当前不承诺完整源码逐文件通读
- 当前不进入部署、运维、性能调优、集群治理的系统化深潜

## Deliverables

### Theme Documents
计划产出以下 7 份文档:

1. `相关概念`
2. `整体架构`
3. `数据写入流程`
4. `向量数据和其他数据的组织格式`
5. `不同数据索引的组织方式`
6. `查询架构`
7. `查询流程串联`

### Document Style
每份文档必须满足:

- 先讲原理
- 再落到本地源码目录
- 明确关键文件
- 给出建议阅读顺序
- 保持能被后续会话直接恢复

## Execution Order

### Theme 1
相关概念与源码入口

### Theme 2
整体架构与组件职责

### Theme 3
数据写入流程与写路径组件

### Theme 4
混合数据组织格式与 schema/field 表达

### Theme 5
索引组织方式与 segment 生命周期关系

### Theme 6
查询架构与查询侧角色边界

### Theme 7
查询流程串联与 end-to-end 路径

## Risks And Controls

### Risk: 版本漂移
控制方式: 固定使用 `2.6.x / v2.6.7` 作为主基线，旧版本差异只做显式备注。

### Risk: 文档事实与源码推断混淆
控制方式: 延续当前写法，把“文档明确结论”和“工作推断”区分开写。

### Risk: 只记目录名，不理解职责
控制方式: 每次先讲“组件为什么存在”，再讲目录和文件。

### Risk: 会话中断导致状态丢失
控制方式: 每个主题完成后立即落盘主题文档，并同步 `task_plan.md`、`findings.md`、`progress.md`。

## Success Criteria
- 用户能按 7 个主题逐步建立 Milvus 心智模型
- 每个主题都能对应到本地源码树中的目录和关键文件
- 新会话能够仅通过项目内文档恢复当前学习进度
- 后续进入源码阅读时，不需要重新发明学习顺序
