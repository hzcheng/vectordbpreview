# Learning Sequence Reorder Design

## Goal
在保持 48 小时主线不变的前提下，调整后续学习顺序，使“尽快形成自己平台的向量数据接入方案”和“系统吃透 Milvus 内部机制”这两个目标兼容，而不是互相阻塞。

## Context
- Phase 1、Phase 2 核心学习、Phase 3 最小 Milvus Lite Demo 已完成。
- 当前原计划是在 Phase 3 之后直接进入平台集成设计，再进入 Milvus 架构与源码深潜。
- 用户提出的顾虑是：如果不先理解 Milvus 的设计实现机制，平台接入方案可能会缺少底层约束，设计不够稳。
- 用户明确的新优先级是：先尽快形成自己平台的接入方案，再回头把 Milvus 的内部机制搞透彻。

## Decision
不将原 `Phase 4` 整体后移到 `Phase 5/6` 之后，而是拆分为两个阶段：

- `Phase 4A`：先完成平台接入方案初稿
- `Phase 5`：阅读 Milvus 架构与数据流，验证方案关键假设
- `Phase 4B`：依据 Milvus 机制修订平台接入方案
- `Phase 6`：只围绕平台方案强相关的问题做源码深潜

## Why This Order

### Option 1: 先完整设计平台方案，再学 Milvus 内核
优点：
- 最快形成可讨论的方案草稿
- 能尽早明确平台视角真正关心的问题

缺点：
- 第一版方案容易过于理想化
- 后续可能因为不了解 Milvus 约束而返工

### Option 2: 先学 Milvus 架构/源码，再设计平台方案
优点：
- 设计时对底层约束理解更扎实

缺点：
- 明显推迟平台方案产出
- 容易先陷入组件细节，缺少平台问题框架
- 可能过度把平台设计绑定到 Milvus 的当前实现

### Option 3: 推荐方案，先出初稿，再用 Milvus 机制校正
优点：
- 最符合当前“先形成平台方案”的优先级
- 读 Milvus 时带着具体问题，学习效率更高
- 既避免平台方案空想，也避免过早陷入内核细节

缺点：
- 需要接受平台方案会经历一次有意识的修订

## Revised Phase Structure

### Phase 2 收尾
- 补一页“场景-问题-方案”对照表
- 目的不是新增学习面，而是关闭已有未完成项

### Phase 4A: 平台接入初稿
- 输出平台接入草图 `v0.1`
- 重点明确：
  - 业务对象与向量化对象
  - schema、主键、metadata
  - embedding 生成时机
  - 更新传播与回流链路
  - 检索入口、过滤、召回与重排边界
  - 权限过滤和多租户边界

### Phase 5: Milvus 架构与数据流验证
- 带着 Phase 4A 的具体问题阅读官方架构和数据流文档
- 核心不是“广泛了解”，而是验证这些平台侧假设：
  - 写入和索引构建是否能解耦
  - segment 生命周期如何影响在线更新
  - filter 与 vector recall 的职责边界
  - 查询 fan-out、冷热数据和 compaction 对平台设计的影响

### Phase 4B: 平台接入方案修订
- 基于 Phase 5 的发现，将平台草图修订为 `v0.2`
- 明确哪些设计直接借鉴 Milvus，哪些是平台层自己的职责

### Phase 6: 聚焦式源码深潜
- 不做无目标源码游览
- 只深入和平台方案直接相关的模块与术语
- 目标是解释关键机制，而不是追求“全量看懂”

## Deliverables
- 一页“场景-问题-方案”对照表
- 一版平台接入草图 `v0.1`
- 一版经 Milvus 机制修订的平台接入草图 `v0.2`
- 一份“平台方案与 Milvus 内部机制映射关系”笔记

## Risks
- 风险：Phase 4A 草图过粗，导致 Phase 5 无法有效验证
  - 缓解：在 Phase 4A 中强制写出关键假设和待验证问题
- 风险：Phase 5 阅读重新变成泛泛扫文档
  - 缓解：阅读时只围绕 `写入`、`索引`、`segment`、`query path`、`filter`、`冷热分层` 六个主题
- 风险：Phase 6 再次失焦
  - 缓解：源码阅读入口由 Phase 4B 中尚未闭环的问题反推

## Success Criteria
- 在不等待源码深潜完成的情况下，先拿到一版可讨论的平台接入方案
- Milvus 架构学习能直接对平台方案产生修订，而不是变成孤立笔记
- 源码阅读目标从“泛化学习”变成“验证平台设计关键点”
